## 一、产品概述

### 1.1 背景

营业部门当前通过手动方式将客户 PDF 指示书转换为内部配色表，再经技术审核后交工厂执行。该流程涉及信息重构、供应商匹配、字段推导等复杂操作，全程依赖人工，效率低且易出错。

BCMS（Box Color Management System）旨在以**配色表**为核心对象，通过 AI 辅助提取 + 人工确认的协同模式，系统化管理箱包配色全流程，从客户 PDF 录入直至打印纸质单交付工厂。

### 1.2 产品目标

- 将客户 PDF → 内部配色表的转换时间从平均 **X 小时缩短至 Y 分钟**（上线后 3 个月内量化基线）
- AI 辅助提取的字段准确率（高置信度字段）≥ 90%
- 支持多人同时操作同一张配色表，无覆盖丢失
- 历史配色表可检索、可复用，翻单操作 5 分钟内完成

### 1.3 第一期核心目标

- 支持 **类型 A**（BEEM 内部生产指示书）和 **类型 B**（外部规范型）两种 PDF 格式的 AI 提取
- 完整的配色表创建、编辑、审核、打印流程
- 历史数据冷启动（Excel 路径优先）
- 翻单（同款换色）功能

> ⚠️ 类型 C（自由型 PDF）暂不支持，待第二期规划。

## 四、核心功能详述

### 4.1 PDF 导入与 AI 提取

#### 4.1.1 上传入口

- 营业在「新建配色表」流程中上传客户 PDF（支持单文件，≤ 50 MB）
- 系统自动识别 PDF 格式类型（A / B），无法识别时提示人工标注
- 上传后进入「提取中」状态，展示进度

#### 4.1.2 AI 提取内容

AI 负责从 PDF 中提取以下信息并输出草稿：

| 提取内容 | 说明 |
|----------|------|
| 款式头信息 | 款号、品名、尺寸、工厂、客户、交期 |
| 颜色变体列表 | 颜色代号及颜色名 |
| 配色行数据 | 部件名、材质、色号、色号备注 |
| 供应商匹配 | 材料代号 → 供应商名（模糊匹配供应商档案库）|
| 字段翻译 | 日文 → 中文（类型 A 专用） |
| 变更说明解析 | 提取编号变更说明（①②③④）并标注可能覆盖的字段 |

AI **不输出**最终配色表，只输出「草稿 + 置信度」。

#### 4.1.3 置信度显示

每个配色行字段标注置信度状态（不显示原始数字）：

| 状态标签 | 计算规则 | 展示颜色 |
|----------|----------|----------|
| 自动通过 | 综合置信度 ≥ 0.85 | 绿色 |
| 需确认 | 0.65 ≤ 置信度 < 0.85 | 橙色 |
| 请核查 | 置信度 < 0.65 | 红色 |
| 推导值 | `is_derived = true` | 灰色（独立标注）|

综合置信度计算公式：`LLM自估 × 格式校验系数 × 原文可溯系数`，详见概念设计文档第十二节。

#### 4.1.4 人工审核界面

- 展示 AI 草稿，逐行高亮「需确认」和「请核查」字段
- 每个字段可查看**来源原文**（`source_text`）
- 营业可直接在界面内修改字段值，修改后该字段标记为「人工修正」
- 全部字段通过后点击「确认提取结果」，进入配色表编辑状态
- 「自动通过」字段可批量确认，无需逐一点击

#### 4.1.5 供应商无法匹配的处理

- 若材料代号在供应商档案库中无法匹配，字段标注「需人工指定供应商」
- 营业可从供应商档案库中手动选择，或新建供应商
- 新建供应商需填写：名称、别名、专项（可选）

---

### 4.2 配色表管理

#### 4.2.1 配色表结构

配色表以「颜色变体」为最小操作单元，一张配色表可包含 N 个颜色变体，每个颜色变体包含 M 条配色行。

配色行字段：`部件大类 / 部件名称 / 材质 / 供应商 / 色号 / 补充说明 / 是否推导值`

#### 4.2.2 动态列

- 每张配色表的部件列由「部件模板」动态决定，不同款式部件组合不同
- 创建配色表时，从预置部件模板中选择（或由 AI 提取结果自动生成列集合）
- 支持在配色表编辑器中增删部件列，增加的部件优先从预置部件库选择，库中无则允许自定义新增
- 列的增删操作记录版本历史

#### 4.2.3 「同色」引用显示与存储

- 数据库存储原始值（如「同色」「面料同色」）
- 界面展示时动态解析为实际色号并在旁边标注来源（如「同色 → 28# 本白」）
- 解析规则：`「同色」/ 「面料同色」/ 「表地同色」→ 引用同行本体色号`；`「各色共通」→ 所有变体此字段相同`
- 打印输出时替换为实际色号，不保留原始引用

#### 4.2.4 悬垂线（省略号）显示

屏幕展示时，多个颜色变体的相同内容用 `〜` 标注：

- 判断依据：同一部件行，不同颜色变体下内容完全一致（「同色」解析后再比较）
- 跨颜色变体比较，不跨款式比较
- 打印输出时恢复完整内容，每行独立完整

#### 4.2.5 版本与历史

- 每次提交修改（由营业或技术操作）自动生成新版本，版本号递增
- 历史版本只读，可查阅、可对比
- 修改记录写入 `color_table_history` 表，包含：操作人、时间、字段名、旧值、新值

---

### 4.3 连锁变更

当主体字段发生变更时，系统自动触发连锁变更提示：

| 主字段变更 | 需确认的关联字段 |
|-----------|----------------|
| 本体色号 | 缝线、包边带、拉链、拉链头、金属件 |
| 内衬色号 | 内衬印花底色 |
| 织带色号 | 织带印花底色 |
| 金属件规格 | 拉链头（如统一改亚镍）|

**交互逻辑**：

1. 营业修改主字段并保存
2. 系统弹出「检测到连锁变更」提示，列出所有受影响字段及其当前值
3. 营业逐一确认是否需要同步修改，或选择「全部同步」
4. 确认结果写入配色行，并标注「连锁更新」来源

**可配置性**：管理员可在后台维护连锁规则表，不同款式类型的规则可独立配置。

---

### 4.4 并发编辑控制

采用**字段级乐观锁**方案：

- 用户打开配色表时，记录当前版本号（`updated_at`）
- 用户 A 提交修改时，服务端检测该字段的版本号是否已变更
- 若已被用户 B 修改，弹出冲突提示：「该字段已被 [用户 B] 修改为 [新值]，请确认」
- 用户 A 选择「保留我的修改」或「使用对方的值」
- 冲突记录写入 `color_table_conflicts` 表，供审计

**说明**：不采用整表锁（阻塞全体操作体验差），不采用字段级悲观锁（实现复杂）。

---

### 4.5 翻单（同款换色）

**适用场景**：新订单与历史订单为同一款式，颜色发生变化。

**操作流程**：

1. 营业点击「新建翻单」，搜索并选择历史款号
2. 系统展示该款所有历史颜色变体（含版本、状态）
3. 营业勾选要复用的颜色变体（支持多选）
4. 系统创建新配色表草稿，复制选中颜色变体的全部配色行
5. 营业在草稿上修改差异部分（色号、供应商、材质等）
6. 修改后触发连锁变更检查（见 4.3）
7. 营业提交，进入审核流程

**版本关联**：新建翻单与来源颜色变体之间建立关联关系，可追溯来源。

---

### 4.6 审核流程

```
[营业] 创建/编辑配色表 → 提交审核
        ↓
[技术] 审阅配色表 → 标注意见（可选）→ 确认通过 / 打回修改
        ↓
[营业] 收到打回通知 → 修改后重新提交
        ↓
[技术] 确认通过 → 配色表状态变为「已确认」
        ↓
[营业] 打印输出，交工厂执行
```

**状态流转**：草稿 → 待审核 → 审核中 → 已确认 / 已打回

---

### 4.7 打印输出

输出规格：

- 纸张：A4 横向
- 所有「同色」引用替换为实际色号
- 悬垂线（〜）恢复为完整内容，每行独立
- 供应商名称完整显示（不用缩写或代号）
- 色号旁附色块（若供应商档案库中有 RGB 值对应）
- 变更说明单独成区，醒目标注
- 页眉包含：款号、品名、版本号、打印日期

---

### 4.8 历史数据导入（冷启动）

#### Excel 导入路径（优先）

1. 管理员/营业上传历史 Excel 文件
2. 系统自动解析列名，展示识别结果及映射建议
3. 用户确认「Excel 列名 → 内部部件字段」的映射关系（支持手动调整）
4. 系统写入 `color_variants` 和 `color_rows`
5. 导入完成后提示抽查确认，支持抽样查看并标记「人工已核验」

**难点处理**：不同营业的 Excel 列名可能不统一，映射确认步骤为必选环节，不可跳过。

#### PDF 历史导入路径（补充）

走完整 AI 提取流程（见 4.1），准确率低于 Excel，作为补充来源。

**冷启动策略**：先跑 Excel 路径，再用 PDF 路径补充覆盖不到的历史订单，不等两条路径同时就绪。

### 4.9 供应商与材料档案管理

管理员功能：

- 供应商 CRUD：名称、别名（多个）、专项分类
- 材料色号台账：材料代号、颜色名、色号 RGB（可选）
- 支持按名称、别名模糊搜索供应商
- 支持批量导入历史供应商数据（CSV）

---

### 4.10 部件与字段映射管理

管理员功能：

- 预置部件分类体系（约 15-20 个大类，可维护）
- 客户字段 → 内部字段映射表，按格式类型（A/B）分别维护
- 连锁变更规则表：主字段 + 关联字段 + 适用款式类型

## 五、非功能性需求

### 5.1 性能

| 指标 | 要求 |
|------|------|
| PDF 提取响应时间 | ≤ 30 秒（类型 A/B，正常页数的 PDF）|
| 配色表页面首屏加载 | ≤ 2 秒 |
| 并发用户数 | 支持 10 人同时在线操作（第一期）|
| Excel 批量导入 | 单文件 500 行以内，≤ 60 秒完成 |

### 5.2 安全与权限

- 登录认证（账号密码，第一期暂不要求 SSO）
- 按角色控制操作权限（营业 / 技术 / 管理员）
- 配色表数据不对外暴露（工厂访问通过打印件，非在线）

### 5.3 可用性

- 系统可用性 ≥ 99%（工作日 8:00–20:00）
- 数据每日备份，支持 7 天内回滚

### 5.4 浏览器兼容性

- Chrome 最新版（主要支持）
- Safari / Edge 最新版（基本兼容）
- 不要求 IE 支持

## 六、数据模型约束

以下约束来自概念设计文档，技术实现必须遵循：

1. **配色行不使用固定列**：必须采用 JSONB 或 EAV 模式存储，配合部件模板动态渲染
2. **「同色」存储原始值**：不在写入时解析，在读取/渲染时动态解析
3. **推导值单独标注**：`is_derived = true` 的字段不参与置信度计算，界面单独标注灰色「推导值」标签
4. **版本号递增**：每次修改 `color_variant` 时 `version` 字段自增，不可跳跃
5. **并发字段**：`ColorRow` 必须包含 `updated_by` 和 `updated_at`，用于乐观锁检测
6. **供应商别名**：`Supplier.alias` 存储为数组，支持多别名模糊匹配
7. **冲突记录不删除**：`color_table_conflicts` 表只写不删，用于审计

## 七、UI/UX 规范

### 7.1 配色表编辑器

- 主视图：以颜色变体为列，部件为行的矩阵表格
- 切换模式：可切换「按变体纵向」和「按部件纵向」两种视图
- 置信度标注：字段背景色（绿/橙/红/灰），不显示数值
- 来源原文查看：点击字段出现 tooltip，显示 `source_text`
- 悬垂线：屏幕模式下显示 `〜`，打印模式下自动展开
- 连锁变更：变更主字段后以 Modal 方式提示关联字段

### 7.2 审核界面

- 「需确认」和「请核查」字段高亮，并在页面顶部显示待确认计数
- 支持「跳到下一个待确认字段」的快捷操作
- 技术审核时可对整个配色表添加备注，不针对单个字段

### 7.3 打印预览

- 打印前必须展示预览页面，预览中显示展开后的完整值
- 支持勾选「仅打印选定颜色变体」

## 十、术语表

| 术语 | 说明 |
|------|------|
| 配色表（Color Table）| 系统第一核心对象，记录款式各部件在不同颜色下的材料、色号、供应商对应关系 |
| 颜色变体（Color Variant）| 系统最小可操作单元，即「款号 + 颜色」，共享款式头信息，各自有独立配色明细 |
| 配色行（Color Row）| 配色表的最小数据单元，记录某颜色变体下某部件的配色信息 |
| 推导值 | 非直接从 PDF 提取，而是由业务规则推导出的配色值（如「缝线 = 面料同色」）|
| 连锁变更 | 主体颜色字段变更时，需要同步确认的关联字段集合 |
| 翻单 | 基于历史颜色变体快速生成新配色表草稿（第一期仅支持同款换色）|
